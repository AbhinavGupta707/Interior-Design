# C7 RoomPlan evaluation

## Current result

The repository contains a deterministic, visibly synthetic conformance split only. It contains no `CapturedRoom`, `CapturedStructure`, `CapturedRoomData`, USDZ, camera frame, world map, customer scan, address or device identifier. On 17 July 2026, `xcrun xctrace list devices` reported the Mac and nine iOS 26.4 Simulators, with no connected physical iPhone or iPad. Physical RoomPlan evaluation is therefore **NOT RUN**.

This is a named C7/C18 release blocker. Simulator navigation, XCUITest fixture journeys, generic iOS compilation and schema fixtures cannot close it.

## What the pack evaluates

The independent evaluator in `tests/evaluation/roomplan` separates three questions:

1. **Synthetic conformance:** strict contracts, safe abstention, immutable source/proposal linkage, zero canonical mutation and bounded resource observations.
2. **Producer integration:** the same observations generated by the integrated API/upload/worker pipeline, explicitly labelled `producer-live`.
3. **Physical field quality:** rights-approved observations from supported LiDAR hardware, with reference measurements and structure-alignment residuals.

Only the first layer is available in this worker. It can find contract and safety defects; it cannot measure RoomPlan accuracy or prove sensor behaviour.

## Rights-controlled split

`tests/evaluation/roomplan/rights-controlled-split.json` is the reviewed split register.

| Split                   |    Current samples | Access                     | Training use | Status       |
| ----------------------- | -----------------: | -------------------------- | ------------ | ------------ |
| `synthetic-conformance` | 12 generated cases | repository-visible         | denied       | available    |
| `physical-development`  |                  0 | restricted evaluation only | denied       | not acquired |
| `physical-holdout`      |                  0 | holdout custodian only     | denied       | not acquired |

Every future physical capture needs its own immutable rights record before inclusion. The record must allow service processing and evaluation for the named purpose; training permission remains separate and denied. Revoked, expired, ambiguous or missing rights exclude the capture and fail closed. Raw physical artifacts stay in the controlled evidence store, not Git. The split register stores opaque evaluation fixture IDs and hashes, never addresses, local paths, object keys, signed URLs, tokens or device advertising identifiers.

Development and holdout properties, rooms and capture sessions must be disjoint. Operators who tune the converter may access only the development split. The holdout custodian releases aggregate observations after the candidate adapter and manifest hash are frozen.

## Running the available evaluation

From the repository root:

```sh
pnpm exec tsc -p tests/evaluation/roomplan/tsconfig.json
pnpm exec vitest run --config tests/evaluation/roomplan/vitest.config.ts
```

The report must retain missing, failed and abstained attempts in their original denominators. A test skip, missing observation, process crash, timeout or resource kill is never removed from coverage.

## Result interpretation

- `passed` means the named gate passed for the evidence class present.
- `failed` means observations exist and violate the deterministic threshold.
- `not-evaluable` means required physical observations do not exist or a sample minimum is unmet. It is never a pass.
- `promotion.eligible` may become true only for a frozen `producer-live` adapter when every gate passes, all physical minimums are met and the field protocol evidence is complete.

See [promotion-rules.md](promotion-rules.md) for exact metrics. Physical execution and evidence capture follow `docs/runbooks/ios/c7-physical-device-field-protocol.md` and `docs/runbooks/ios/c7-release-evidence-template.md`.
