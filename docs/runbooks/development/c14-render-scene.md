# C14 deterministic render-scene builder

## Scope and truth boundary

`@interior-design/render-scene` converts one exact succeeded C10 scene into the frozen
`c14-render-scene-manifest-v1` document. It does not create or modify geometry. C4/C5 remain
canonical dimensional truth, the immutable C10 GLB remains protected derived geometry, and C13
remains authoritative for selected product/material pins and rights.

The package is deliberately declarative. It accepts bytes and validated records in memory and
returns canonical JSON bytes plus an external SHA-256 envelope. It never accepts or emits a
`.blend` file, Python, a driver/expression, a URI, a filesystem path, an environment-variable
reference, or executable content. It performs no environment lookup, network access, subprocess
launch, renderer discovery, or Blender capability probe.

No Blender command is part of this runbook. The C14 hardware render gate remains deferred to an
authorised render host.

## Required server-resolved input

Call `buildRenderScene` only after the server has resolved all of the following authoritative
records; client-supplied hashes are not authority:

1. one succeeded C10 `SceneJob`, its `SceneRecord`, exact GLB bytes, and C10 manifest/artifact
   hashes;
2. the exact C4 canonical snapshot referenced by that C10 manifest;
3. one working C13 specification whose current immutable revision targets that exact snapshot;
4. the published C13 catalog release, its exact canonical release-manifest bytes, and the complete
   set of asset-version records named by the release;
5. an explicit canonical C4 camera ID and integer near/far clip planes;
6. a frozen C14 render profile and renderer-script SHA-256.

The package validates, rather than assumes, the entire chain. It recomputes the C10 manifest and
GLB hashes, C4 snapshot hash, C13 revision hash, catalog release-manifest hash, asset-version hashes,
placement-projection hashes, and all cross-record project/model/snapshot/release/specification
references. A release must be active and every referenced catalog asset must remain approved for
service processing, commercial use, derivatives, and rendered-output distribution. Training
permission remains denied.

## GLB validation and double binding

The protected input must be a bounded canonical GLB 2.0 file generated by
`interior-design-scene-compiler/1.0.0`. The parser enforces the C10 52,428,800-byte public artifact
ceiling, a 32 MiB JSON-chunk ceiling, bounded graph/accessor counts, exact chunk lengths/alignment,
finite Float32 accessors, in-range triangle indices, internal buffers, supported C10 extensions,
one-owner canonical node mappings, and exact C10 manifest counts.

The parser rejects malformed/truncated containers, non-canonical or duplicate-key JSON,
NaN/infinity, external resources, URI fields, external images, animations, skins, sparse accessors,
unsafe extensions/extras, executable-like keys/text, paths, and additional chunks.

C13 is verified twice:

- `asset.extras.c13SpecificationBinding` must be present with exactly the authoritative
  specification revision and catalog release pins; and
- every C13 line must have exactly one matching `c13CatalogBinding` on its C10 node or material,
  with no missing, duplicate, forged, stale, or extra element binding.

The C10 manifest alone is insufficient because it intentionally does not carry this C13 binding.

## Deterministic output

The builder sorts materials, canonical lights, protected element IDs, findings, and segmentation
mappings with code-unit comparison; it never uses locale collation, wall-clock time, timezone, or
randomness. Canonical JSON uses the repository I-JSON/JCS implementation. The manifest's
`determinismKeySha256` hashes its deterministic source and configuration inputs, not its own bytes.

The returned external envelope is:

```ts
{
  schemaVersion: "c14-render-scene-external-sha256-v1";
  manifestSchemaVersion: "c14-render-scene-manifest-v1";
  byteLength: number;
  sha256: string;
}
```

The manifest contains neither this SHA-256 nor an artifact record for its own bytes. Always persist
or publish the canonical bytes returned by `canonicalBytes()` and store the envelope separately.
The method returns a fresh byte copy on every call.

## Coordinate, camera, material, light, and world policy

- C4 and Blender are right-handed and Z-up. `c4MillimetresToBlenderMetres` divides exact safe
  integer millimetres by 1,000 without axis permutation; 1 mm maps to 0.001 m.
- `deriveBlenderCamera` validates the frozen C14 clip/FOV contract and constructs a deterministic
  local-camera-to-world basis where local `-Z` is forward. The normal world-up is canonical `+Z`.
  A vertical view uses the named `positive-y-world-up-fallback-v1`; coincident position/target is
  rejected.
- Canonical point lights retain exact integer luminous flux and colour temperature. The named
  `c14-photometric-to-blender-v1` conversion is pinned at 683 lumens per watt. Spot, linear, and
  area lights are omitted with a finding because C4 lacks the required cone/orientation/shape
  parameters. `daylight-reference` is also omitted and never creates a sun or environment.
- One supported C13 material retains its sRGB8, metallic/roughness basis points, emissive values,
  asset-version hash, rights hash, and sorted texture artifact hashes. Unsupported material sets,
  UV sets, or missing UVs use the fixed neutral `[153,153,153]` fallback with a finding. A colour is
  never derived from an asset hash.
- The world is always `neutral-studio-no-address-or-daylight-inference-v1`.
- Segmentation uses sorted mesh-owning canonical element IDs. The palette encodes an index as
  `[floor(index/65536)+1, floor(index/256)%256, index%256]`, keeping black reserved and remaining
  collision-free for the frozen 100,000-element manifest limit.

The pure camera/coordinate/photometric helpers are exported so the renderer lane can apply exactly
the same named conversion without changing the declarative manifest.

## Safe failures and logging

Failures throw `RenderSceneError`. Its `code` and `diagnostic()` result are safe for structured
telemetry; messages are static and never interpolate IDs, paths, hashes, catalog text, URLs, source
bytes, or request content. Do not log caught causes, input records, the GLB JSON, manifests, artifact
bytes, specification notes, rights/licence text, or storage locators.

Expected safe codes include `SOURCE_HASH_MISMATCH`, `CANONICAL_SOURCE_MISMATCH`,
`C10_BINDING_MISMATCH`, `C13_BINDING_MISMATCH`, `C13_RIGHTS_DENIED`, `GLB_INVALID`,
`GLB_RESOURCE_LIMIT`, `GLB_EXTERNAL_RESOURCE`, `GLB_UNSAFE_CONTENT`, `CAMERA_INVALID`, and
`MANIFEST_INVALID`.

## Renderer-lane integration

The C14-L2 worker should:

1. server-resolve and hash-verify the inputs above before claim-time staging;
2. call `buildRenderScene` in memory and persist its external envelope separately from the canonical
   manifest bytes;
3. stage only the already-verified C10 GLB and canonical manifest in a fresh private workspace;
4. use the exported coordinate/camera/photometric policies in its fixed renderer driver;
5. treat `protectedElementIds` and `segmentationPalette` as validation inputs, never as permission
   to alter C10 geometry; and
6. publish renderer/output artifacts only behind the separate C14 lease and publication fence.

The renderer must not add a silent CPU fallback, daylight inference, address context, pseudo-product
appearance, or a canonical mutation. The safe render remains independent of optional enhancement.

## Verification

From the repository root, without invoking Blender:

```sh
pnpm --filter @interior-design/render-scene lint
pnpm --filter @interior-design/render-scene typecheck
pnpm --filter @interior-design/render-scene test:unit
pnpm --filter @interior-design/render-scene build
git diff --check
```

The unit suite uses byte fixtures only. It covers exact hashes/bindings, forged and stale pins,
withdrawn rights, malformed/truncated/duplicate-key/external/unsafe/non-finite/oversize GLBs, 1 mm
mapping, clip/FOV/camera degeneracy and collinearity, neutral UV/material fallback, 100,000-colour
palette collision checks, stable sorting/key ordering, privacy-safe diagnostics, and byte-identical
fresh processes under different locales and timezones.

These checks prove deterministic scene construction and safe input handling. They do not constitute
Blender, renderer, GPU, provider, perceptual-quality, or real-artifact evidence.
