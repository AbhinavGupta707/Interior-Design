# C6 Contract — Floor-plan proposal, calibration and correction

## Status and outcome

- Checkpoint: C6
- Contract versions: `c6-plan-job-v1`, `c6-plan-parser-input-v1`, `c6-plan-proposal-v1`, `c6-plan-operation-draft-v1`
- Frozen from: C5 ledger-close commit `f86b72a`
- Outcome: a rights-cleared, C2-approved floor-plan asset produces a typed and source-pinned proposal or an explicit abstention. An authorised user can inspect the source/proposal overlay, provide scale evidence, accept/correct/exclude candidates and submit exact C5 operations through preview and commit while residual unknowns and correction effort remain visible.
- Scope boundary: C6 understands one bounded plan page. It does not claim a survey, infer structure/fire/regulatory compliance, silently invent occluded geometry, fuse RoomPlan/reconstruction evidence (C7-C9), render 3D (C10), train on customer uploads or let parser output mutate canonical state directly.

## Resolved DQ-018 and supported input box

1. Development and the required gate use no paid provider, credential, outbound inference call or GPU. The adapter boundary remains versioned so an evaluated learned model can be added later without changing public job/proposal semantics.
2. The supported v1 source box is a `ready` C2 asset with `kind=plan`, retained service-processing consent and one of `application/pdf`, `image/svg+xml`, `image/png` or `image/jpeg`. A job selects exactly one zero-based page from the first 20 pages, limits the source to 25 MiB and raster work to 20 megapixels, and refuses any source fingerprint/status/rights mismatch.
3. The geometric box is one level per selected page, straight wall centrelines, hosted rectangular door/window openings and closed straight-edged room boundaries. Curves, freehand sketches, perspective photographs, reflected ceiling plans, sections/elevations, multi-page registration, stairs, furniture/object recognition, structure, annotations as instructions and full-house fusion are out of box and must remain unknown or cause abstention.
4. Processing is vector-first: a no-credential spatial sandbox converts approved PDF/SVG vector content into a bounded normalized manifest. Raster PDF/image fallback is rewritten to a metadata-free grayscale input and handled by a deterministic CPU line/opening baseline. Text is untrusted label data only and can never alter tool policy.
5. The deterministic fixture adapter is always available for local/CI evidence. A parser/model adapter can be promoted only behind the same schema after rights, licence, holdout, severe-error, calibration and abstention evidence beats the deterministic baseline.

## Security and processing contract

- C2 upload/signature/hash/content validation remains mandatory. C6 rechecks tenant/project/asset identity, `ready` state, exact source SHA-256, rights and training denial before leasing work.
- Source processing runs without platform/cloud credentials or outbound network, with generated filenames, isolated temporary storage, bounded output, a 30-second parser deadline, at most three attempts and deterministic cleanup. Public/workflow records contain IDs, hashes, versions, counts and safe codes—not raw file bytes, object keys, signed URLs, extracted text, local paths or parser stderr.
- PDF/SVG/raster tools are version-recorded. SVG DTD/entity/script/style/event/external-resource constructs remain rejected. Parser output is strict-schema parsed, bounded, source-scoped and geometry validated before publication.
- Job creation/retry/calibration/operation-draft mutations require an 8–128 character `Idempotency-Key`. The same actor/action/key/body replays one result; any actor/action/body mismatch conflicts. Cancellation is monotonic and cannot erase a completed proposal or audit history.
- Owner/editor may create/cancel/retry/calibrate/draft; viewer may read jobs/proposals only. Every request reauthenticates current membership and checks the project before asset/job/proposal disclosure. Machine parser identity may propose but never calibrate, draft, preview or commit.
- Job and result production are durable and transactional. Crash/retry yields one terminal result for the exact asset/page/parser/input hash. A cancelled or superseded lease cannot publish. Unsupported, unsafe, malformed, resource-exhausting, invalid-output and low-confidence cases terminate with bounded abstention/failure codes.

## Frozen proposal, calibration and operation semantics

1. A normalized parser input pins job/project/asset/page, source hash/MIME/coordinate space, normalizer/tool versions and resource limits. Vector/raster/fixture modes emit the same proposal-result union.
2. A proposal contains stable UUID candidates for a level, walls, hosted openings and spaces; integer source coordinates; candidate/overall confidence; source regions; parser version/manifest hash; findings; and explicit unresolved regions. It never contains a raw source, secret, storage locator or executable instruction.
3. Confidence is an evaluation signal, not truth. Candidate confidence below 60 is unresolved; overall confidence below 75, a severe geometry finding, invalid hosting/topology, output truncation or source mismatch forces abstention. Partial proposals may display only if every omitted/uncertain region is explicit and no operation draft can hide it.
4. Calibration retains human/source evidence and a bounded rational affine transform from integer source coordinates to integer millimetres. Known-length evidence includes both selected source points and the asserted millimetre length. Mapping uses exact integer arithmetic and a documented half-away-from-zero rounding rule. No valid calibration means overlay-only review and no operation draft.
5. Candidate decisions are exactly `accepted`, `corrected`, `excluded` or `unresolved`. Corrections retain the original candidate ID and resulting C5 client-operation IDs. Exclusion/unresolved decisions never silently create geometry.
6. An operation draft pins proposal/calibration, target branch revision/head hash and 1–50 exact public C5 operations. It records correction/exclusion/unresolved counts, elapsed review milliseconds and acknowledged warnings. The platform reparses operations and verifies candidate mapping, calibration, topology and current access; it does not commit.
7. The user explicitly sends the returned draft through the existing C5 preview and commit endpoints. C5 remains the only canonical mutation boundary, revalidates geometry and concurrency, and records immutable operation/audit/outbox history. Parser candidates can never call a database/state tool directly.
8. Source, normalized-input and proposal hashes remain inspectable after commit. Proposal and calibration records are immutable; a correction creates a new operation draft. Retry creates a new attempt against the same pinned source or a new job when input/parser selection changes.

## Frozen job states, routes and recovery

States are `queued`, `processing`, `proposed`, `abstained`, `cancel-requested`, `cancelled` and `failed`. Only queued/processing work can be cancelled. Only retryable abstained/failed work can be retried, at most three attempts. Terminal proposal and abstention are immutable.

| Method       | Route                                                                           | Permission / behavior                                  |
| ------------ | ------------------------------------------------------------------------------- | ------------------------------------------------------ |
| `GET`/`POST` | `/v1/projects/:projectId/plan-processing-jobs`                                  | list readable jobs / create from one exact ready asset |
| `GET`        | `/v1/projects/:projectId/plan-processing-jobs/:jobId`                           | tenant-scoped status and safe diagnostics              |
| `POST`       | `/v1/projects/:projectId/plan-processing-jobs/:jobId/cancel`                    | owner/editor monotonic cancellation                    |
| `POST`       | `/v1/projects/:projectId/plan-processing-jobs/:jobId/retry`                     | owner/editor bounded retry                             |
| `GET`        | `/v1/projects/:projectId/plan-processing-jobs/:jobId/proposal`                  | immutable proposal or explicit abstention              |
| `POST`       | `/v1/projects/:projectId/plan-processing-jobs/:jobId/proposal/calibrations`     | owner/editor immutable scale/transform evidence        |
| `POST`       | `/v1/projects/:projectId/plan-processing-jobs/:jobId/proposal/operation-drafts` | owner/editor typed C5 draft, never direct commit       |

Stable recovery actions are replace source, select another page, add a known dimension, retry a retryable tool failure, use the manual C5 editor or request professional/survey input. A failed spinner without an explicit status/next step is forbidden.

## Provisional benchmark and promotion thresholds

- The repository benchmark is synthetic/rights-cleared evaluation evidence, not a real-home or survey accuracy claim. Every fixture has a source hash, origin/creator, licence/right, allowed purpose, split and training-use status; holdout fixtures cannot be imported by parser implementation code.
- All failures and abstentions stay in denominators. The v1 gate requires at least 90% accepted-input coverage on declared in-box fixtures, 100% abstention on declared hard negatives, zero severe errors and zero cross-tenant/source/rights violations.
- On accepted calibrated synthetic plans: wall endpoint P90 error is at most 50 mm, opening-centre P90 at most 75 mm and calibration residual P90 at most 25 mm. A severe error is any wrong level, unhosted opening, invalid/self-intersecting room, hidden omitted region, source mismatch or wall endpoint error over 250 mm presented as acceptable.
- Confidence expected-calibration error must be at most 0.15 only when a split has at least 20 evaluable confidence samples; smaller runs report `insufficient-sample` and cannot support promotion. Risk/coverage and error-by-confidence band are always reported.
- Processing reports median/P90 wall time, CPU and peak-memory observations without asserting production scale. The hard local deadline is 30 seconds/page. The provisional supported-plan correction target is median at most 8 minutes and P90 at most 15 minutes; CI proves instrumentation/action counts, while the time target remains `not-measured` until a rights-approved human study exists and cannot be reported as passed from automated timing.

## Frozen correction experience

- The project/evidence flow offers only `ready` plan assets and shows source filename, rights basis, page and immutable SHA prefix. Starting a job explains supported input and that the result is a proposal, not a survey.
- The plan workspace has an accessible source/proposal overlay, opacity control, source/proposal toggle, zoom/pan, level/candidate list and structured keyboard inspector. Raw SVG/PDF is never injected into the DOM; only safe derived raster/vector primitives render.
- Known-length calibration is keyboard and pointer accessible, displays source points, asserted length, scale and residual, and requires confirmation. Changing calibration creates new evidence/draft state rather than silently moving committed geometry.
- Candidate review exposes confidence, source region, inferred/unknown status and accepted/corrected/excluded/unresolved decision. Users can correct wall endpoints/thickness, opening host/position/width and space name/boundary through structured controls backed by exact integer values.
- Before C5 preview the review summary shows typed operation count, corrected/excluded/unresolved counts, warnings/acknowledgements, target revision/head and the fact that commit is still reversible only through later compensating history. Errors and unresolved severe regions block preview.
- Responsive acceptance covers 1440×960 and 390×844, keyboard-only completion, refresh/re-authentication, cancellation/retry, explicit abstention/manual fallback and stale C5 conflict recovery. Status changes use live regions and no core task depends on canvas-only interaction.
