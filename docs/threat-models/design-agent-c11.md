# C11 bounded design-agent threat model

**Status:** implementation threat model for C11-L2

**Scope:** `@interior-design/model-gateway` and the platform `design-agent` module

**Data/provider posture:** deterministic synthetic fixtures and `deterministic-local-v1` only; every external provider is disabled
**Last reviewed:** 2026-07-18

## Security objective

The design agent may read a bounded projection of the current C11 draft brief and one household message, then return an expiring, typed C11 patch proposal, clarification or professional-review route. The model gateway cannot access a database or object store, read a filesystem, call a generic network tool, update a brief, mutate a canonical home/model branch or make a professional determination. The platform module has a narrow Postgres repository for C11 consultation sessions/messages/proposals and shared C11 idempotency effects only; it has no C4/C5/C9/C10 mutation path.

Confirmation is a separate authorised application action. It reloads and revalidates the tenant, project, session, proposal, expiry, base brief identity and exact revision, validates every patch operation again, and invokes only the injected `BriefCommandPort.confirmProposal`. L1's `PostgresBriefRepository.confirmProposal` is the single transaction owner for the idempotency claim, brief revision, confirmed proposal state, linkage/audit record and completed session. L2 does not implement a second confirmation transaction. The proposal remains non-mutating until L1's command succeeds.

This follows the NCSC principle that security is a lifecycle requirement and that providers should own downstream security outcomes, use the most secure option by default, and disclose limitations. The NCSC also identifies prompt injection as an AI-specific attack class. See [NCSC Guidelines for secure AI system development — Introduction](https://www.ncsc.gov.uk/collection/guidelines-secure-ai-system-development/introduction).

## Assets and sensitive data

| Asset                                     | Sensitivity and required treatment                                                                                                                                                                                             |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Household consultation message            | Private project data; bounded to 8,000 characters; persisted only through the tenant-scoped repository port; never a metric/log attribute                                                                                      |
| Accessibility or health-related statement | Potential UK GDPR special-category data; minimise to the functional design need, keep local, never log, never use for training and never send externally                                                                       |
| Inferred health/accessibility status      | Do not infer a diagnosis or clinical conclusion; if processing intentionally infers health status, treat it as special-category processing regardless of confidence                                                            |
| Current brief projection                  | Tenant/project scoped; only stable IDs, classifications, categories, statuses and bounded statements reach the local adapter                                                                                                   |
| Evidence excerpt                          | Untrusted document data; bounded and never interpreted as prompt/tool policy; the C11 platform bridge currently supplies none                                                                                                  |
| Proposal                                  | Immutable payload with append-only state events; exact session/project/base revision/source message/provider registry versions; expires after 30 minutes; a newer turn rejects older pending heads as `superseded-by-new-turn` |
| Correlation and safe telemetry            | Safe codes, adapter/prompt/tool versions, counts and durations only; no raw input/output, locator, token, secret or health/accessibility content                                                                               |

The ICO explains that health data includes information about disability and any data revealing physical or mental health status. It also states that intentionally inferred health status is special-category processing regardless of confidence. See [ICO: What is special category data?](https://ico.org.uk/for-organisations/uk-gdpr-guidance-and-resources/lawful-basis/special-category-data/what-is-special-category-data/). Local processing is a risk reduction, not a lawful-basis decision: production activation with identifiable household data still needs the organisation's documented UK GDPR lawful basis, Article 9 condition where applicable, retention policy and DPIA assessment.

## Trust boundaries and data flow

1. One of the six exact Fastify consultation routes authenticates, authorises the frozen C11 action, verifies project membership/scope and passes an actor plus tenant-safe correlation context to the design-agent service. Every response is `private, no-store`; write headers and body/client-message idempotency keys must agree.
2. The design-agent service loads the exact draft brief and session through injected ports. Foreign-tenant/project lookups return not found.
3. The service constructs the fixed `model-gateway-request-v1` envelope. The caller cannot choose a prompt or tool: the only prompt is `c11-consultation-extract-v1`, and the only tool is the side-effect-free `c11.propose-brief-patch-v1` tool.
4. The model gateway enforces exact keys, character/item/timeout limits, an exact adapter registry and cancellation. `deterministic-local-v1` performs deterministic local extraction. `external-disabled` always returns a safe unavailable code and has no network path.
5. The platform treats the result as untrusted, rejects unknown keys/tool calls/malformed operations, and validates it against the frozen C11 contracts before persistence.
6. Explicit confirmation repeats scope, lifecycle, expiry and revision validation and calls the injected brief-command port with an idempotency key and expected revision. The result contains the full committed frozen `DesignBrief`, proposal and session so the route performs no follow-up mutation. No C4/C5/C9/C10 mutation port exists in this module.

## Threats and controls

| Threat                                   | Attack/failure                                                                                                     | Preventive and detective controls                                                                                                                                                                                                                                                                                                             | Residual risk / response                                                                                                                                                                    |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Prompt or document injection             | User/evidence text says to ignore policy, reveal prompts or invoke another tool                                    | Untrusted text occupies data-only fields; exact prompt/tool registries; suspicious instruction-like user text produces clarification; evidence excerpts are never interpreted as policy; result schema rejects tool calls                                                                                                                     | Novel instruction wording may look like an ordinary preference. It still cannot select a tool or create side effects. Continue red-team corpus expansion.                                   |
| Tool smuggling / confused deputy         | Request or result asks for HTTP, SQL, filesystem, object storage or canonical mutation                             | Strict unknown-key rejection at request and result boundaries; only `entry.add` proposal operation; capability manifest explicitly denies generic network/filesystem/database/storage/brief/canonical mutation; static architecture tests                                                                                                     | Future registry expansion is security-sensitive and requires a new frozen contract and threat-model review.                                                                                 |
| Malformed or overprivileged model output | Wrong schema, duplicate IDs, unknown fields, unsupported operation, forged manifest or mismatched request ID       | Gateway result validation plus independent platform Zod/C11 validation; fixed provider manifest; request-ID match; safe `MODEL_INVALID_OUTPUT` / `DESIGN_AGENT_INVALID_MODEL_OUTPUT` errors                                                                                                                                                   | A defect in both validators could admit bad data; C11 brief repository must validate again atomically.                                                                                      |
| Resource exhaustion                      | Oversized messages/excerpts/context/output, too many operations/turns or slow execution                            | Envelope byte/character/item limits; C11 shared limits; maximum 20 model operations; 2-second platform timeout within a 5-second gateway maximum; abort propagation; 100-turn session cap                                                                                                                                                     | CPU denial within accepted bounds remains possible; deployment should add authenticated rate/concurrency limits at the orchestrator-owned route layer.                                      |
| Cancellation race                        | Client/session cancellation races local extraction or persistence                                                  | Abort signal; safe cancellation code; repository append requires active session and exact turn count; cancellation/confirmation repository commands use expected lifecycle and turn count                                                                                                                                                     | The L1 durable implementation must transact lifecycle checks with append/confirm records.                                                                                                   |
| Tenant/project IDOR                      | Foreign IDs retrieve a session/proposal or confirm another project                                                 | Every repository/read-command method is tenant and project scoped; service cross-checks returned IDs; foreign scope fails as not found before disclosure                                                                                                                                                                                      | Composition must preserve server-side authz for all eight C11 actions; route-policy matrix is orchestrator/L3 evidence.                                                                     |
| Cross-actor replay                       | Another editor reuses a visible/guessed client message ID and body to receive another actor's original effect      | Immutable message `created_by`; stored-turn read model includes `createdByUserId`; service and transactional append require the same actor and content hash; mismatches return one generic conflict without content/effect disclosure                                                                                                         | Authorised project collaborators can still submit their own new message IDs; rate and anomaly controls remain deployment responsibilities.                                                  |
| Stale/replayed confirmation              | Expired/stale/older proposal or reused key mutates a newer brief                                                   | Proposal TTL; at most one pending head per session; every newer turn atomically rejects older pending heads; exact session/proposal/base checks; expected revision; single L1 confirmation idempotency ledger; same-key/different-body conflict                                                                                               | Root composition must adapt exactly one L1 confirmation port; any second confirmation transaction or nested idempotency owner is forbidden.                                                 |
| Direct canonical mutation                | Agent writes C4/C5/model/database state                                                                            | Model gateway has no SQL capability; L2 Postgres repository is limited to consultation lifecycle/idempotency tables and never writes brief revisions; only L1-backed `BriefCommandPort.confirmProposal` may revise a brief; static tests prohibit model-operation/canonical imports                                                           | A future composition could inject an overbroad port. The root adapter must expose only L1's C11 brief read and atomic confirmation operations.                                              |
| Professional overclaim                   | Assistant answers structural, regulatory, clinical-accessibility, fixed-cost, live-stock or professional questions | Deterministic review classifier returns typed `review-required`; review-only proposal cannot be confirmed as a patch; generic limitations in user-facing summary                                                                                                                                                                              | Regex classification can miss paraphrases. Unrecognised ambiguity produces clarification rather than an answer; independent policy/comprehension evaluation remains required.               |
| Special-category disclosure              | Health/accessibility data appears in telemetry, errors or an external request                                      | External disabled; no training; local-only manifest; functional minimisation; generic safe errors; telemetry contains counts/duration/safe code only; redaction tests include secrets and health text                                                                                                                                         | Raw messages are necessarily available to the tenant-scoped repository. Encryption, access audit, retention/deletion and incident response are L1/orchestrator deployment responsibilities. |
| False provider capability                | Fixture/local output is presented as external or network-backed                                                    | Exact `deterministic-local-v1` manifest with `externalNetworkUsed: false`; `external-disabled` reports unavailable; no external result can satisfy the C11 proposal manifest                                                                                                                                                                  | UI must retain this honest status and must not relabel fixture/local results.                                                                                                               |
| Weak audit or hidden limitation          | Incidents cannot be reconstructed, or users over-trust output                                                      | Correlation context at command boundaries; proposal captures exact source message and registry versions; confirmation linkage records actor/time/brief revision; safe transition metrics; explicit limitations and review routes                                                                                                              | Raw prompts are intentionally absent from general logs. Authorised incident access to tenant records must be separately controlled and audited.                                             |
| Derived-ID collision or ambiguity        | Attacker crafts two source tuples that receive one entry ID                                                        | Entry IDs are RFC 9562 UUIDv8 values from the first 128 SHA-256 bits over ordered u32-be-length-prefixed UTF-8 frames: namespace `interior-design:c11:brief-entry:rfc9562-v8:sha256:v1`, source message ID, classification, category and exact bounded statement; standard variant/version bits; determinism/separation/collision-smoke tests | Truncation leaves a theoretical 122-bit collision space after UUID bits. A detected duplicate is rejected by brief-domain uniqueness checks and must be treated as a safe conflict.         |

The NCSC secure-deployment guidance recommends access controls, query-interface protections, incident procedures, high-quality audit support, evaluation/red-teaming before release, secure defaults and transparent limitations. These controls are reflected in the exact registries, bounded query/result interfaces, lifecycle audit linkage, adversarial tests and disabled-by-default provider posture. See [NCSC Guidelines — Secure deployment](https://www.ncsc.gov.uk/collection/guidelines-secure-ai-system-development/guidelines/secure-deployment).

## Safe error and telemetry contract

Errors expose only a stable safe code, generic message/status and retryability where relevant. Causes, raw messages, prompts, model output, accessibility/health content, asset locators, credentials and tokens are not serialised. Metrics contain only:

- lifecycle stage and outcome;
- `deterministic-local-v1` or `external-disabled`;
- safe error code;
- duration; and
- operation, clarification and professional-review counts.

Tenant ID, project ID and user/message content are deliberately excluded from metric labels.

## Security verification

Automated C11-L2 tests cover:

- prompt and evidence-document injection;
- request/result tool smuggling and unknown keys;
- malformed manifests/results and unsupported operations;
- message, context, result and turn resource bounds;
- timeout and caller cancellation;
- conflict and explicit unknown preservation;
- all professional-review reasons;
- stale, expired, cancelled, cross-project and cross-tenant proposals;
- same-actor replay, cross-actor denial, create replay before mutable brief checks and same-key/different-body conflict;
- old-proposal supersession and the at-most-one-pending invariant;
- all six production routes, strict response schemas, role/IDOR/no-store/header-body checks;
- guarded real-Postgres ordering, replay, cancellation/rejection, expiry, isolation, confirmation lookup, tamper and rollback cases;
- external-disabled honesty with a zero-network assertion;
- redaction-safe errors and telemetry; and
- RFC 9562 UUIDv8 determinism, framed-name separation and adversarial collision smoke coverage; and
- static and behavioural proof that the model boundary and consultation repository have no direct canonical mutation path.

## Open deployment requirements

- Root composition must adapt `BriefCommandPort.findCurrent(tenantId, projectId, briefId) -> DesignBrief | undefined` to L1's current revision record and adapt `BriefCommandPort.confirmProposal(command)` to L1's `BriefProposalConfirmationPort.confirmProposal` by renaming only `command.request` to `command.update`. The full `{ brief, proposal, replayed, session }` result passes through unchanged. L1 remains the only confirmation transaction/idempotency owner.
- The orchestrator-owned `services/platform-api/src/c11.ts` must instantiate `PostgresDesignAgentRepository`, the L1-backed brief adapter, deterministic model-gateway bridge and `DesignAgentService`, then call `registerDesignAgentRoutes`. Central composition must retain the frozen role/action matrix, rate/concurrency limits and machine-principal confirmation denial.
- Production data governance must define the lawful basis, any Article 9 condition, retention/deletion, access audit and DPIA decision for health/accessibility data.
- Independent red-team, multi-browser workflow and user-comprehension evidence are required before C11 closes.
- External providers remain disabled. Enabling one requires a new provider/data/rights/security decision, egress controls, contract change and reassessment; this threat model does not authorise it.
